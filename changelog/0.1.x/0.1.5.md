# 0.1.5 — 2026-07-16

## Added

- **`nps_find_events` `occurrenceDates` / `isRecurring`** — each event gains `occurrenceDates`, the record's `dates[]` intersected with the requested `dateStart`/`dateEnd` window (the full remaining set when no window is given), plus `isRecurring`. NPS returns `dates[]` independent of the query window, so a recurring series rendered only its anchor `dateStart`/`dateEnd` could show dates outside the window — the new fields carry the dates that actually fall in it ([#2](https://github.com/cyanheads/national-parks-mcp-server/issues/2)).
- **`nps_find_events` `invalid_state_code` error reason** — the tool accepted a `stateCode` input but had no matching error-contract entry, so a bad state code fell through to a generic failure. It now declares `invalid_state_code` with the same recovery hint as the other state-filtered tools ([#3](https://github.com/cyanheads/national-parks-mcp-server/issues/3)).

## Fixed

- **`parkCode` / `stateCode` recovery hints now reach the client** — format validation moved off the Zod input schema into the handler in `nps_find_parks`, `nps_get_alerts`, `nps_find_campgrounds`, and `nps_find_events`. A schema-level `.regex()` failure threw a raw `ZodError` before the tool's declared `recovery` existed, so an invalid code never returned its recovery hint; the handler check surfaces it ([#3](https://github.com/cyanheads/national-parks-mcp-server/issues/3)).
- **`nps_find_events` rejects impossible calendar dates** — a shape-valid but non-existent date like `2026-02-31` passed the `YYYY-MM-DD` regex and reached the upstream 400. A synchronous `Date.UTC` round-trip check now rejects it client-side with the `invalid_date` recovery hint ([#8](https://github.com/cyanheads/national-parks-mcp-server/issues/8)).

## Dependencies

- **`tsc-alias`** `^1.9.0` → `^1.9.1`
